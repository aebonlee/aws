### 기초 모델(FM) 성능 향상 방법

- AI 모델의 성능 향상 가이드: 기초부터 고급 전략까지
    
    <aside>
    💡
    
    여러분은 혹시 외국어를 배워본 경험이 있으신가요? 처음에는 단어장을 외우고, 나중에는 원어민과 대화하면서 실력을 키우고, 때로는 영화나 드라마를 보면서 자연스럽게 배우기도 하죠. 놀랍게도 AI도 이와 매우 비슷한 방식으로 '공부'를 합니다. AI가 어떻게 더 똑똑해지는지, 그 흥미로운 방법들을 함께 알아보겠습니다.
    
    ### 1. **AI의 맞춤형 학습: 기초 모델 커스터마이징**
    
    가장 기본이 되는 방법은 기존의 잘 훈련된 AI 모델을 우리의 목적에 맞게 수정하는 것입니다. 이는 마치 기성복을 나에게 맞게 수선하는 것과 비슷합니다.  이 방법에는 두 가지 주요 접근법이 있습니다.
    
    - **1.1. 지속적 사전 학습 (Continued Pre-training)**
        
        전문 분야의 자료를 계속 읽게 하는 방식입니다. 예를 들어, 의료 AI를 만들고 싶다면 의학 논문과 교과서를 계속 읽게 합니다. 레이블(정답)이 없는 데이터로도 가능하며, 시간이 지날수록 해당 분야에 대한 이해도가 자연스럽게 높아집니다. 
        
        ```
        의료 분야 특화 AI를 만든다고 가정해 봅시다.
        - 기존 AI에게 의학 교과서, 논문들을 계속 '읽게' 합니다
        - 처음에는 "위궤양"이 무슨 말인지 몰랐지만...
        - 점점 의학 용어와 개념을 이해하게 됩니다
        - 마치 의대생이 공부하면서 성장하는 것과 비슷하죠!
        ```
        
    - **1.2. 정밀 조정 (Fine-tuning)**
        
        더 직접적인 학습 방법으로, 문제와 정답이 있는 데이터로 훈련시키는 방식입니다. 두 가지 방식이 있습니다.
        
        - 지시사항 기반(Instruction-based): "이렇게 하세요"라고 직접 알려주는 방식
        - 도메인 적응(Domain adaptation): 특정 분야의 전문성을 키우는 방식
    
    ### 2. 즉시 활용형 접근법
    
    기존 모델을 수정하지 않고도 더 나은 결과를 얻을 수 있는 방법들도 있습니다.
    
    - **2.1. 맥락 학습 (In-context Learning)**
        
        마치 새로운 게임을 할 때 튜토리얼을 보는 것처럼, 몇 가지 예시만 보여주고 바로 활용하는 방식입니다.
        
        ```
        사용자: "이 영화 어때요?"
        AI 예시1: "재미있어요" → 긍정
        AI 예시2: "별로예요" → 부정
        ```
        
        이런 예시들을 보여주면, AI는 비슷한 패턴으로 응답할 수 있게 됩니다.
        
    - **2.2. 프롬프트 기반 학습 (Prompt-based Learning)**
        
        명확한 지시사항으로 원하는 결과를 이끌어내는 방식입니다. 예를 들어, "전문가처럼 자세하게 설명해 주세요" 같은 지시를 주면 그에 맞게 응답합니다.
        
    - **2.3. RAG (검색 증강 생성, Retrieval Augmented Generation)**
        
        마치 오픈북 시험처럼, AI가 필요한 정보를 찾아보고 답하는 방식입니다. 최신 정보나 정확한 사실 확인이 필요할 때 특히 유용합니다.
        
    
    이러한 다양한 방법들은 각자의 장단점이 있습니다. 
    
    - 사전 학습은 시간이 오래 걸리지만 깊이 있는 이해가 가능합니다.
    - 맥락 학습은 빠르지만 복잡한 문제에는 한계가 있을 수 있습니다.
    - RAG는 정확한 정보를 제공하지만 추가 리소스가 필요합니다.
    
    상황과 목적에 따라 적절한 방법을 선택하거나 여러 방법을 조합하는 것이 중요합니다. 마치 학습자가 자신에게 맞는 공부 방법을 찾아가는 것처럼, AI도 최적의 학습 방법을 통해 더 나은 성능을 달성할 수 있습니다.

    - 학생 모델이 교사 모델의 출력 패턴을 학습하여, 적은 파라미터로도 유사한 성능을 달성

    - 모델 수정(학습) 없이 입력 형식만 조정
    - **최소한의 구현 노력으로 가능** |
    |  |  | RAG
    (Retrieval Augmented Generation) | - 외부 지식/문서를 검색하여 모델 응답에 통합
    - 최신 정보 반영 + 도메인 특화 응답 가능
    - 응답에 참조된 데이터 링크를 포함하여 응답에 대한 신뢰도를 향상시킬 수 있음
    - RAG 파이프라인: 데이터 준비 + 검색 및 생성

       - 임베딩 및 인덱싱: 분할된 각 청크를 **벡터로 변환(임베딩 생성)**하고, 검색 가능한 **벡터 인덱스에 저장(서치 인덱스 구축)**. 새 콘텐츠가 추가될 때마다 주기적으로 실행
     - **검색 및 생성(실시간 처리)**
       - 사용자 질문이 들어오면 이를 **임베딩**으로 변환
       - 인덱스에서 **관련성/ 높은 청크를 검색**(Retrieval)
       - 검색된 청크와 사용자 질문을 함께 LLM에 전달해 최종 **답변을 생성**(Generation) |

    - 각 외부 서비스마다 독자적인 API 통합을 구축할 필요 없이, MCP를 통해 통합 연결
    - 모델이 정보에 기반한 의사결정에 필요한 추가 컨텍스트를 확보할 수 있게 함

        **벡터 검색의 본질적 개념**
        
        벡터 검색은 데이터의 의미를 이해하고 유사성을 기반으로 검색하는 시스템입니다. 전통적인 검색이 단순히 키워드를 매칭하는 것과 달리, 벡터 검색은 데이터에서 자동으로 의미를 추출하여 비슷한 항목을 찾아냅니다.
        
        **개발 프로세스의 주요 단계**
        
        개발 프로세스는 크게 네 단계로 이루어 집니다.
        
        첫째, 데이터 전처리 단계입니다. 데이터는 벡터 검색 시스템의 기반이 되며, 시스템의 성능을 좌우하는 핵심 요소입니다. 온라인 쇼핑몰의 경우 상품 이미지, 상품명, 상품 설명 등이 주요 데이터가 됩니다. 이 단계에서는 데이터 수집, 정제, 변환 등이 수행됩니다. 예를 들어, 청바지 상품의 경우,
        
        - 청바지 상품 이미지: 배경을 제거하고, 크기를 표준화
        - 청바지 상품명 텍스트: ‘2024 봄 신상 워싱 와이드 청바지’ → 핵심 키워드 추출(예. ‘와이드 청바지’)
        - 가격, 사이즈 등 메타 데이터를 일정한 형식으로 정리(정규화)
        
        둘째, 벡터 변환 단계입니다. 이 단계는 정제된 데이터를 벡터로 변환하는 과정입니다. 
        
        - 표준화된 청바지 상품 이미지: 청바지의 시각적 특징(핏, 워싱, 색상 등)을 [0.2, 0.8, 0.5, ...] 같은 숫자 배열로 변환
        - 핵심 키워드가 추출된 텍스트: ‘와이드 청바지’라는 텍스트를 [0.7, 0.3, 0.9, ...] 형태로 변환
        - 결합 벡터: 이미지와 텍스트 정보를 모두 고려하여 하나의 통합된 벡터 생성 (예. [0.2, 0.8, 0.5, ...]  + [0.7, 0.3, 0.9, ...] )
        
        셋째, 유사도 검색 단계입니다. 사용자가 특정 상품에 관심을 보일 때, 벡터 간의 거리를 계산하여 가장 유사한 상품들을 추천합니다.
        
        - 사용자가 ‘연청 와이드 청바지’ 상품 페이지 방문
        - 해당 상품의 벡터와 가장 가까운 거리에 있는 다른 상품들을 찾음
        - ‘같은 핏의 다른 워싱’, ‘비슷한 스타일의 다른 브랜드’ 등을 추천
        
        넷째, 시스템 개선 단계입니다. 실제 사용자 데이터를 기반으로 시스템을 지속적으로 개선합니다. 예를 들어, 추천 상품의 클릭률, 실제 구매 전환율, 사용자 피드백 등을 분석하여 시스템을 지속적으로 개선합니다.
        
        **AWS의 벡터 스토어 서비스**
        
        AWS는 다양한 벡터 검색 솔루션을 제공합니다. 
        
        1. Amazon DocumentDB는 대규모 문서형 데이터를 저장하고 검색하는데 특화되어 있습니다. 블로그 포스트, 제품 설명서, 고객 리뷰 등을 효율적으로 관리할 수 있어 콘텐츠 관리 시스템에 적합합니다.
        2. Amazon **OpenSearch** Service는 실시간 검색과 분석이 필요한 경우에 활용됩니다. 특히 전자상거래 사이트의 상품 검색이나 대규모 로그 분석 시스템에서 많이 사용됩니다. 사용자의 검색 의도를 정확하게 파악하고 관련성 높은 결과를 신속하게 제공할 수 있습니다.
        3. **Amazon RDS for PostgreSQL**에는 pgvector 확장을 통해 벡터 검색 기능이 추가되었습니다. 기존 관계형 데이터베이스를 사용하면서도 벡터 검색의 이점을 활용할 수 있어, 사용자 프로필 기반의 추천 시스템 구축에 적합합니다.
        4. **Amazon Aurora PostgresSQL**는 고성능 관계형 데이터베이스로, 벡터 검색 기능과 일반적인 데이터베이스 기능을 함께 활용할 수 있습니다. 금융 거래 패턴을 분석하거나 이상 거래를 탐지하는 등 복잡한 데이터 처리가 필요한 시스템에 적합합니다.
        
        **활용 분야와 효과**
        
        벡터 검색은 전자상거래의 상품 추천뿐만 아니라, 넷플릭스의 콘텐츠 추천, 의료 영상 비교 분석, 금융 거래 패턴 분석 등 다양한 분야에서 활용됩니다. 온라인 쇼핑몰의 경우, 고객이 원하는 상품을 쉽게 찾을 수 있게 되고, 구매 전환율과 고객 만족도가 향상되는 효과를 얻을 수 있습니다.
        
        이처럼 벡터 검색은 단순한 키워드 매칭을 넘어, 데이터의 의미를 이해하고 활용함으로써 더 스마트한 서비스를 가능하게 합니다.

- LLM/FM 평가 방법론
    
    <aside>
    💡
    
    LLM/FM 평가의 큰 흐름 이해하기
    
    **LLM 평가 방식의 두 가지 유형**
    
    LLM을 평가하는 방법은 크게 두 가지로 나눌 수 있습니다. 인간 평가(Human Evaluation)와 자동 평가(Automated Evaluation)입니다
    
    - **인간 평가(Human Evaluation)**
    인간 평가는 사람이 직접 모델이 생성한 응답을 보고 평가하는 방식으로, 질적인 측면에서 평가가 가능하지만 비용과 시간이 많이 소요됩니다. 평가 항목으로는 문장의 자연스러움(유창성), 신뢰성(정확성), 사용자 질문 관련성, 공정성 등이 있습니다. LLM/FM 평가에서 가장 중요한 방식이며 최종적으로는 사람이 선호하는 모델을 선택해야 합니다. 다만 인간 평가는 평가자마다 일관성이 부족할 수 있고, 비용이 많이 들며, 자동화하기 어렵다는 단점이 있습니다.
    - **자동 평가(Automated Evaluation)**
    자동 평가는 알고리즘을 사용하여 모델의 성능을 자동으로 측정하는 방식입니다. 수치화가 가능하여 빠르게 비교할 수 있지만, 인간이 느끼는 품질을 완전히 반영하기는 어렵습니다. 대표적인 평가 지표로는 번역 및 요약 성능을 평가하는 BLEU/ROUGE, 다음 단어 예측 정확도를 측정하는 Perplexity(혼잡도), 생성 문장과 정답 문장의 유사도를 BERT 모델로 평가하는 BERTScore 등이 있습니다. 자동 평가는 빠르고 일관된 비교가 가능하며 수치화된 결과를 얻을 수 있지만, 실제 인간 만족도와는 차이가 있을 수 있습니다.
        - **BLEU/ROUGE**: 번역 및 요약 성능 평가 (출력과 정답 비교)
        - **Perplexity(혼잡도)**: 다음 단어를 얼마나 정확하게 예측하는지 측정
        - **BERTScore**: 생성된 문장과 정답 문장의 유사도를 BERT 모델로 평가
    
    FM(Foundation Model) 평가는 "인간 평가"와 "자동 평가"를 어떻게 효과적으로 활용할지가 중요합니다. 평가 단계별 흐름은 네 단계로 이루어집니다.
    
    **FM 평가의 단계별 프로세스**
    
    - **1단계: 기본 자동 평가 실시**
    기본적인 자동 평가를 수행합니다. 모델이 BLEU, ROUGE, Perplexity 등 특정 지표에서 높은 점수를 받는지 확인하고, 낮은 점수를 받는다면 문법 오류나 문맥 파악 실패 같은 기본적인 문제가 있는지 분석합니다.
    - **2단계: 인간 평가단 샘플링 평가**
    커스텀 프롬프트 데이터셋을 활용하여 인간 평가단을 통한 샘플링 평가를 수행합니다. 자동 평가에서 높은 점수를 받은 모델들 중 일부 샘플을 선정하여 사람이 직접 평가하고, 실제 사용자가 원하는 응답 스타일과의 일치 여부, 창의성, 공정성, 논리적 정합성 등을 점검합니다.
    - **3단계: 종합 평가 및 모델 선택**
    사전 정의된 벤치마크 데이터셋으로 모델의 정확도를 측정하고, 자동 평가와 인간 평가를 종합하여 최적의 모델을 선택합니다. 자동 평가 점수가 낮아도 인간 평가가 높은 경우는 창의적인 응답일 가능성이 있고, 자동 평가 점수는 높지만 인간 평가가 낮은 경우는 기계적으로 답변하지만 사용자 친화적이지 않을 가능성이 있습니다.
    - **4단계: 지속적 개선 및 피드백 반영**
    실제 사용자 피드백을 수집하여 지속적으로 모델을 개선합니다. 필요한 경우 파인튜닝(fine-tuning)을 수행하여 모델 성능을 향상시킵니다.
    
    ### **3. 인간 평가의 중요성과 필수 영역**
    
    - **창의적 표현 평가:** 창의적인 작업에서는 BLEU/ROUGE 같은 자동 지표로는 창의적인 답변을 제대로 평가할 수 없어 인간 평가가 필수적입니다.
    - **맥락 이해력 평가:** NLP 모델의 맥락 이해 능력은 같은 단어를 사용해도 문맥에 따라 의미가 달라질 수 있어 인간의 세밀한 판단이 필요합니다.
    - **윤리성 및 공정성 평가:** AI의 윤리적 응답과 편향 없는 공정성 평가에는 인간의 사회적, 문화적 판단이 중요한 역할을 합니다.
    </aside>
    
- LLM 성능 평가를 위한 지표와 데이터셋
    
    
    | 지표 또는 데이터셋 | 설명   |
    | --- | --- |
    | F1 Score | 1. **분류 모델**의 성능을 평가하는 지표로, 정밀도(Precision)와 재현율(Recall)의 조합 지표(**균형**)
    2. LLM 평가에도 활용: 텍스트 생성 결과를 이진 분류 문제로 변환. **정확한 토큰 매칭 기반**
     - 예측(모델의 답변)이 정답 참조(ground truth)와 얼마나 일치하는지를 단어/구절 단위로 평가
     - QA(Question Answering): 예측 답변과 정답 간 토큰 단위 F1 계산
     - NER/정보 추출: 엔티티나 키워드 추출의 정확성 평가 |
    | BERTScore | 1. **의미적 유사성 평가**(BERT 임베딩 모델을 활용)
     ****- 생성된 문장과 참조 문장의 의미가 얼마나 비슷한가(임베딩 코사인 유사도 측정)  |
    | **B**LEU
    (Bilingual Evaluation Understudy) | 1. 기계 **번역 평가 지표** + **텍스트 요약 평가**
    2. n-gram **정확 매칭. 정확도 중심**(**B**y the book, 책대로, 정확히)
     - 생성된 텍스트(모델의 번역)가 참조 번역 텍스트(사람의 번역) 기준으로 얼마나 정확한가 |
    | ROUGE
    (**Recall**-Oriented Understudy for Gisting Evaluation) | 1. 텍스트 **요약 평가 지표** + 기계 **번역 평가**
    2. n-gram **중첩 측정. 재현율 중심**(기억해냈나, 포함했나)
     - 생성된 텍스트가 참조 텍스트의 내용을 얼마나 포함했나 |
    | GLUE
    (General Language Understanding Evaluation) | 텍스트 분류, 질의응답, 자연어 추론 등 일반적인 언어 이해 작업 평가를 위한 데이터셋 모음 |
    | SuperGLUE | GLUE를 확장한 것으로, 더 도전적인 작업과 복합적 언어 이해에 초점 |
    | SQuAD
    (Stanford Question Answering Dataset) | 질의 응답 능력 평가용 데이터셋 |
    | WMT(Workshop on Machine Translation) | 기계 번역 시스템 평가를 위한 데이터셋과 작업들 |
    | Perplexity(복잡도) | 모델이 다음 토큰을 얼마나 잘 예측하는지, 주어진 단어 시퀀스를 생성할 확률을 측정 |
    | MMLU
    (Massive Multitask Language Understanding) |  대규모 다중 작업 언어 이해
     LLM의 지식과 추론 능력을 평가하기 위한 벤치마크로, 다양한 학문 분야(의학, 수학, 역사, 법률, 윤리 등)에 걸친 57개의 과목에서 추출한 문제들로 구성 |
    | BIG-bench | 현재 모델 능력을 넘어서는 작업 평가 |
    | HELM
    (Holistic Evaluation of Language Models) | 언어 모델의 전체론적 평가
    단순한 성능 측정을 넘어서 모델의 다양한 측면(정확성, 견고성, 공정성, 편향성, 투명성, 효율성 등)을 종합적으로 평가 |

### FM 평가의 고급 방법론

- LLM-as-a-judge (LLM을 평가자로 사용)
    - **LLM 자체를 평가 도구로 활용**하여 다른 모델이나 에이전트의 응답 품질을 자동으로 평가하는 방식
    - Amazon Bedrock에서 기본 제공하는 평가 메트릭
        
        
        | 메트릭 | 설명 |
        | --- | --- |
        | Completeness (완전성) | 응답이 프롬프트의 **모든 부분을 얼마나 철저하게** 다루는지 평가 |
        | Following instructions (지침 준수) | 응답이 **특정 지침이나 제약 조건을 준수**하는지 평가 |
        | Refusal (거부) | 모델이 부적절한 프롬프트에 대해 **적절하게 응답을 거부**하는지 평가 |
    - 인간 평가보다 빠르고 비용이 적게 들며, 프로덕션 환경에서 대규모 자동 평가에 적합
- A/B 테스트
    - 통제된 환경에서 **실제 사용자**에게 두 가지 이상의 모델 변형(A, B)을 노출시키고 결과를 비교하는 방법
    - **실제 애플리케이션 성능**을 가장 직접적으로 측정할 수 있는 전략
    - 인간 평가(전문가 품질 판단)와의 차이: A/B 테스트는 실제 사용자의 행동 데이터(클릭률, 전환율 등)를 기반으로 평가
- FM 평가 방법 종합 비교
    
    
    | 평가 방법 | 평가 주체 | 측정 대상 | 적합 상황 |
    | --- | --- | --- | --- |
    | 자동 평가 (BLEU, ROUGE 등) | 알고리즘 | 텍스트 품질 | 빠른 모델 비교 |
    | 인간 평가 | 전문가 | 응답 품질, 창의성, 공정성 | 주관적 판단 필요 시 |
    | LLM-as-a-judge | LLM | 완전성, 지침 준수, 거부 | 프로덕션 대규모 평가 |
    | A/B 테스트 | 실제 사용자 | 사용자 행동, 비즈니스 결과 | 실전 성능 검증 |

### 기초 모델(FM) 상황별 대응 방법

| 상황 | 방법 |
| --- | --- |
| 기존 코드를 다른 언어로 변환하기 위해 생성형 AI를 사용할 때 고려해야 하는 기준 | 구문, 의미 이해 및 코드 최적화 기능 |
| 복잡한 하드코딩된 로직을 시장과 제품 라인 전반에 걸쳐 확장할 때, 생성형 AI 모델이 제공하는 장점 | 적응성(Adaptability) |
| 신약에 대한 사용자 리뷰를 분석하고 각 약물에 대한 간략한 개요를 제공 | Amazon Bedrock 대규모 언어 모델(LLM)을 사용 |
| 챗봇이 대규모 연구 논문 데이터베이스를 참고해 답변 생성중. 연구 논문에 사용된 복잡한 과학 용어로 인해 FM의 성능이 저조한 문제 발생 | 도메인 적응 미세 조정(domain adaptation fine-tuning) 적용 |
| 레이블이 지정된 데이터셋에서 AI 모델을 훈련하는 기법 | Fine-tuning |
| 명령 기반 파인튜닝을 사용할 때 훈련 데이터를 준비하는 방법 | 도메인과 관련된 주제를 구체적으로 다루는 질문과 답변 쌍 생성 |
| 회사의 데이터를 사용하여 모델을 더 정확하게 **미세 조정(fine-tuning)** | 프롬프트 필드 및 완료 필드와 함께 **레이블이 지정된 데이터**를 제공 |
| 회사가 원하는 톤에 맞게 챗봇의 응답을 개선하고자 함. 상담원과 고객 간의 고품질 대화 100개 예시를 보유. 이 데이터를 활용하여 챗봇의 응답에 회사 톤을 반영하기 위한 솔루션 | Bedrock으로 파인튜닝 |
| LLM 출력이 짧고 특정 언어로 작성되길 희망 | 프롬프트를 수정하는 것으로 충분 |
| 챗봇이 회사 분위기에 맞는 응답 생성하길 희망 | 원하는 응답을 생성할 때까지 실험하고 프롬프트를 수정 |
| 생성된 콘텐츠가 회사의 브랜드 톤앤매너와 메시징 요구사항에 부합하도록 보장 | 명확한 지시사항과 맥락을 제공하는 효과적인 프롬프트를 작성하는 것으로 충분 |
| 몇 가지 구체적인 예시를 제공하여 에이전트의 정확도를 향상하기 위한 방법 | 예시를 포함하도록 고급 프롬프트 수정 |

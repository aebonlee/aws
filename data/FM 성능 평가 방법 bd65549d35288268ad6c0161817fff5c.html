기초 모델(FM) 성능 향상 방법
  • AI 모델의 성능 향상 가이드: 기초부터 고급 전략까지
  • 성능 향상 방법 분류
    분류 설명종류방법
    FM
    커스터마이징    		모델을 직접 수정하는 것:
    - 가중치를 조정
    - 새로운 데이터 추가
    그 결과로 새로운 모델이 생성    		Continued(Continuous)
    Pre-training    		- 레이블 없는 데이터로 도메인 지식 학습(Domain adaptation with unlabeled data)
    - 일반 → 특정 분야(의료, 법률, 금융) 적응
    - 가장 비싼 방법
    - 지속적인 사전 학습(ongoing pre-training)을 통해 시간이 지날수록 모델의 성능이 향상됨
    Fine-tuning- 레이블이 지정된 데이터로 특정 작업 학습
    - 모델을 커스터마이징 하는 방법 중에서는 비교적 비용 효율적
    - 두 가지 방식:
    - Instruction-based fine-tuning: 프롬프트-응답 쌍(pair)을 통해 추가 학습
    - Domain adaptation fine-tuning: 레이블된 데이터로 도메인 특화 학습(도메인 지식, 용어 등)
    Distillation
    (모델 증류)    		- 더 크고 성능이 좋은 교사(teacher) 모델의 지식을 더 작은 학생(student) 모델로 전달하는 기법
    - 학생 모델이 교사 모델의 출력 패턴을 학습하여, 적은 파라미터로도 유사한 성능을 달성
    - 교사 모델이 레이블을 만들어주는 Fine-tuning
    - 교사 모델 대비 추론 비용 절감 및 지연 시간 단축
    FM 정렬
    (Alignment)    		모델의 행동을 인간의
    가치와 선호에 맞게 조정
    - 가중치를 조정
    - 성능 향상 X. 행동 정렬이 목적    		RLHF
    (Reinforcement Learning from Human Feedback)
    		- 숙련된 인간 평가자의 선호도 피드백을 보상 신호로 활용하여 (학습 단계에서) 모델 행동을 최적화
    - 새로운 지식을 가르치는 것이 아니라, 이미 아는 것을 어떻게 표현할지를 조정
    - 회사의 가치, 윤리, 톤앤매너에 맞게 모델을 정렬(alignment)
    - 새로운 트렌드나 새로운 유형의 문제에 지속적으로 적응 가능
    - 예1. 콘텐츠 조정(Content Moderation): 회사 정책에 맞는 콘텐츠 필터링
    - 예2. 챗봇 톤 조정: 긍정적 고객 피드백에 보상을 제공하여 응대 품질 향상
    FM 활용모델 자체는 변하지 않음
    (학습 X, 튜닝 X)    		In-context Learning- 프롬프트 내 구체적 예시 제공(few shot learning)
    - 문제와 정답의 여러 쌍을 제공
    - 예. 사용자의 입력의 의도를 잘 파악하게 하기 위해서, 아래의 쌍을 함께 전달
    - (사용자 메시지1, 사용자 의도1)
    - (사용자 메시지2, 사용자 의도2)
    - 최소한의 구현 노력으로 가능
    Prompt-based learning- 프롬프트 엔지니어링으로 원하는 결과 유도
    - 명령어, 지시사항 중심
    - 모델 수정(학습) 없이 입력 형식만 조정
    - 최소한의 구현 노력으로 가능
    RAG
    (Retrieval Augmented Generation)    		- 외부 지식/문서를 검색하여 모델 응답에 통합
    - 최신 정보 반영 + 도메인 특화 응답 가능
    - 응답에 참조된 데이터 링크를 포함하여 응답에 대한 신뢰도를 향상시킬 수 있음
    - RAG 파이프라인: 데이터 준비 + 검색 및 생성
    - 데이터 준비: 계산 집약적이므로 오프라인/배치 처리
    - 청킹(Chunking): 원본 문서를 의미 있는 작은 단위(chunk)로 분할. 검색 시 질문과 관련성 높은 컨텍스트(contextual relevancy)를 정확히 찾아내기 위한 과정
    - 임베딩 및 인덱싱: 분할된 각 청크를 벡터로 변환(임베딩 생성)하고, 검색 가능한 벡터 인덱스에 저장(서치 인덱스 구축). 새 콘텐츠가 추가될 때마다 주기적으로 실행
    - 검색 및 생성(실시간 처리)
    - 사용자 질문이 들어오면 이를 임베딩으로 변환
    - 인덱스에서 관련성/ 높은 청크를 검색(Retrieval)
    - 검색된 청크와 사용자 질문을 함께 LLM에 전달해 최종 답변을 생성(Generation)
    MCP
    (Model Context Protocol)    		- FM이 외부 데이터 소스 및 서비스에 표준화된 방식으로 연결할 수 있게 하는 개방형 표준(open standard)
    - 각 외부 서비스마다 독자적인 API 통합을 구축할 필요 없이, MCP를 통해 통합 연결
    - 모델이 정보에 기반한 의사결정에 필요한 추가 컨텍스트를 확보할 수 있게 함
    - 예. 여행 예약 에이전트가 항공사 API, 호텔 예약 시스템 등 외부 서비스에 MCP로 연결
    • 벡터 검색(Vector search) 이해하기
  • LLM/FM 평가 방법론
  • LLM 성능 평가를 위한 지표와 데이터셋
    지표 또는 데이터셋설명
    F1 Score1. 분류 모델의 성능을 평가하는 지표로, 정밀도(Precision)와 재현율(Recall)의 조합 지표(균형)
    2. LLM 평가에도 활용: 텍스트 생성 결과를 이진 분류 문제로 변환. 정확한 토큰 매칭 기반
    - 예측(모델의 답변)이 정답 참조(ground truth)와 얼마나 일치하는지를 단어/구절 단위로 평가
    - QA(Question Answering): 예측 답변과 정답 간 토큰 단위 F1 계산
    - NER/정보 추출: 엔티티나 키워드 추출의 정확성 평가
    BERTScore1. 의미적 유사성 평가(BERT 임베딩 모델을 활용)
    - 생성된 문장과 참조 문장의 의미가 얼마나 비슷한가(임베딩 코사인 유사도 측정)
    BLEU
    (Bilingual Evaluation Understudy)    		1. 기계 번역 평가 지표 + 텍스트 요약 평가
    2. n-gram 정확 매칭. 정확도 중심(By the book, 책대로, 정확히)
    - 생성된 텍스트(모델의 번역)가 참조 번역 텍스트(사람의 번역) 기준으로 얼마나 정확한가
    ROUGE
    (Recall-Oriented Understudy for Gisting Evaluation)    		1. 텍스트 요약 평가 지표 + 기계 번역 평가
    2. n-gram 중첩 측정. 재현율 중심(기억해냈나, 포함했나)
    - 생성된 텍스트가 참조 텍스트의 내용을 얼마나 포함했나
    GLUE
    (General Language Understanding Evaluation)    		텍스트 분류, 질의응답, 자연어 추론 등 일반적인 언어 이해 작업 평가를 위한 데이터셋 모음
    SuperGLUEGLUE를 확장한 것으로, 더 도전적인 작업과 복합적 언어 이해에 초점
    SQuAD
    (Stanford Question Answering Dataset)    		질의 응답 능력 평가용 데이터셋
    WMT(Workshop on Machine Translation)기계 번역 시스템 평가를 위한 데이터셋과 작업들
    Perplexity(복잡도)모델이 다음 토큰을 얼마나 잘 예측하는지, 주어진 단어 시퀀스를 생성할 확률을 측정
    MMLU
    (Massive Multitask Language Understanding)    		 대규모 다중 작업 언어 이해
    LLM의 지식과 추론 능력을 평가하기 위한 벤치마크로, 다양한 학문 분야(의학, 수학, 역사, 법률, 윤리 등)에 걸친 57개의 과목에서 추출한 문제들로 구성
    BIG-bench현재 모델 능력을 넘어서는 작업 평가
    HELM
    (Holistic Evaluation of Language Models)    		언어 모델의 전체론적 평가
    단순한 성능 측정을 넘어서 모델의 다양한 측면(정확성, 견고성, 공정성, 편향성, 투명성, 효율성 등)을 종합적으로 평가
FM 평가의 고급 방법론
  • LLM-as-a-judge (LLM을 평가자로 사용)
    • LLM 자체를 평가 도구로 활용하여 다른 모델이나 에이전트의 응답 품질을 자동으로 평가하는 방식
    • Amazon Bedrock에서 기본 제공하는 평가 메트릭
      메트릭설명
      Completeness (완전성)응답이 프롬프트의 모든 부분을 얼마나 철저하게 다루는지 평가
      Following instructions (지침 준수)응답이 특정 지침이나 제약 조건을 준수하는지 평가
      Refusal (거부)모델이 부적절한 프롬프트에 대해 적절하게 응답을 거부하는지 평가
    • 인간 평가보다 빠르고 비용이 적게 들며, 프로덕션 환경에서 대규모 자동 평가에 적합
  • A/B 테스트
    • 통제된 환경에서 실제 사용자에게 두 가지 이상의 모델 변형(A, B)을 노출시키고 결과를 비교하는 방법
    • 실제 애플리케이션 성능을 가장 직접적으로 측정할 수 있는 전략
    • 인간 평가(전문가 품질 판단)와의 차이: A/B 테스트는 실제 사용자의 행동 데이터(클릭률, 전환율 등)를 기반으로 평가
  • FM 평가 방법 종합 비교
    평가 방법평가 주체측정 대상적합 상황
    자동 평가 (BLEU, ROUGE 등)알고리즘텍스트 품질빠른 모델 비교
    인간 평가전문가응답 품질, 창의성, 공정성주관적 판단 필요 시
    LLM-as-a-judgeLLM완전성, 지침 준수, 거부프로덕션 대규모 평가
    A/B 테스트실제 사용자사용자 행동, 비즈니스 결과실전 성능 검증
기초 모델(FM) 상황별 대응 방법
상황방법
기존 코드를 다른 언어로 변환하기 위해 생성형 AI를 사용할 때 고려해야 하는 기준구문, 의미 이해 및 코드 최적화 기능
복잡한 하드코딩된 로직을 시장과 제품 라인 전반에 걸쳐 확장할 때, 생성형 AI 모델이 제공하는 장점적응성(Adaptability)
신약에 대한 사용자 리뷰를 분석하고 각 약물에 대한 간략한 개요를 제공Amazon Bedrock 대규모 언어 모델(LLM)을 사용
챗봇이 대규모 연구 논문 데이터베이스를 참고해 답변 생성중. 연구 논문에 사용된 복잡한 과학 용어로 인해 FM의 성능이 저조한 문제 발생도메인 적응 미세 조정(domain adaptation fine-tuning) 적용
레이블이 지정된 데이터셋에서 AI 모델을 훈련하는 기법Fine-tuning
명령 기반 파인튜닝을 사용할 때 훈련 데이터를 준비하는 방법도메인과 관련된 주제를 구체적으로 다루는 질문과 답변 쌍 생성
회사의 데이터를 사용하여 모델을 더 정확하게 미세 조정(fine-tuning)프롬프트 필드 및 완료 필드와 함께 레이블이 지정된 데이터를 제공
회사가 원하는 톤에 맞게 챗봇의 응답을 개선하고자 함. 상담원과 고객 간의 고품질 대화 100개 예시를 보유. 이 데이터를 활용하여 챗봇의 응답에 회사 톤을 반영하기 위한 솔루션Bedrock으로 파인튜닝
LLM 출력이 짧고 특정 언어로 작성되길 희망프롬프트를 수정하는 것으로 충분
챗봇이 회사 분위기에 맞는 응답 생성하길 희망원하는 응답을 생성할 때까지 실험하고 프롬프트를 수정
생성된 콘텐츠가 회사의 브랜드 톤앤매너와 메시징 요구사항에 부합하도록 보장명확한 지시사항과 맥락을 제공하는 효과적인 프롬프트를 작성하는 것으로 충분
몇 가지 구체적인 예시를 제공하여 에이전트의 정확도를 향상하기 위한 방법예시를 포함하도록 고급 프롬프트 수정
텍스트의 감정을 긍정이나 부정으로 분류프롬프트에 긍정 또는 부정 레이블이 있는 텍스트 예시들을 제공하고, 그 뒤에 분류할 새 텍스트를 배치
조작되어 바람직하지 않은 작업을 수행하거나 민감 정보 노출 방지LLM에 공격 패턴을 탐지하도록 지시하는 프롬프트 템플릿 제공
질문을 하는 사람에 따라 생성형 AI 모델의 응답 스타일을 자동으로 변경하길 희망. 최소한의 구현 노력으로 사용자의 연령대 정보를 모델에 제공하길 희망모델 응답이 목표로 해야 할 연령대를 모델에게 지시하는 역할 설명을 프롬프트 컨텍스트에 추가
예. "8살 아이에게 설명하듯이 답변해 주세요”
회사 직원이 선호하는 스타일로 응답을 생성하는 모델을 탐색 중인력과 맞춤형 프롬프트 데이터셋을 사용하여 모델을 평가
복잡한 자연어 처리 작업(예: 텍스트 요약)의 경우, 모델의 품질과 성능을 평가하는 데
인간 평가자는 문맥의 이해, 논리적 흐름, 의미의 정확성 등을 종합적으로 판단
인간 평가자들은 생성된 요약문의 일관성, 유창성, 전반적인 품질에 대해 주관적인 판단을 제공할 수 있으며, 이는 자동화된 평가 지표(예. 단순한 정확도나 수치적 지표)만으로는 파악하기 어려운 부분
빠르게 변화하는 재고 데이터를 사용중. 모델의 정확도를 향상시켜야 함검색 증강 생성(Retrieval Augmented Generation) 사용
대형 언어 모델(LLM)과 지식 베이스를 사용하여 텍스트 기반 챗봇 인터페이스 구현Amazon Bedrock을 사용하여 Retrieval Augmented Generation (RAG) 에이전트 개발
인사 정책에 대한 질문에 답하는 LLM 챗봇 개발중. 대규모 디지털 문서를 보유. 챗봇의 응답을 최적화하기 위해 회사가 사용해야 하는 기법검색 증강 생성(RAG)
제품 매뉴얼을 바탕으로 고객 문의에 답변하는 AI 기반 에이전트의 응답에 대한 고객 신뢰도를 향상시킬 수 있는 전략응답에 참조된 제품 매뉴얼 링크 포함
회사의 내부 데이터 소스에서 가져온 관련 데이터를 사용하여 FM 모델 보완Amazon Bedrock 지식 베이스를 생성
FM이 회사 정보를 사용하여 더 많은 컨텍스트를 포함하게 하는 방법Bedrock Knowledge Bases를 사용
회사 제품 설명서(PDFs)를 위한 채팅 인터페이스를 개발. 비용 효율적인 방법을 고민중Amazon Bedrock 지식 베이스에 PDF 문서를 업로드. 지식 베이스를 사용하여 사용자가 프롬프트를 제출할 때 컨텍스트로 제공(RAG)
미해결 고객 청구건을 확인하고, 특정 청구건에 대한 세부사항을 파악하며, 청구건 관련 문서에 접근할 수 있도록 도와주는 AI 애플리케이션을 개발Amazon Bedrock 지식 베이스와 함께 Amazon Bedrock용 에이전트를 사용하여 애플리케이션을 구축
RAG 기반 솔루션. 사용자에게 거의 실시간 경험을 제공하길 희망. 오프라인 배치 처리를 사용하여 구현해야 하는 RAG 파이프라인의 단계콘텐츠 임베딩 생성, 검색 인덱스 생성
검색 애플리케이션이 텍스트와 이미지가 포함된 쿼리를 처리멀티모달 임베딩 모델
Amazon Bedrock을 사용한 결과 및 주제가 어린이에게 적합한지 확인Amazon Bedrock Guardrails 사용
모델이 응답에 개인 환자 정보를 포함하는 것을 방지하고 위반 시 알람Amazon Bedrock Guardrails을 사용하여 콘텐츠 필터링 + Amazon CloudWatch를 사용해 알람
챗봇이 과거의 상호작용과 온라인 리소스를 통해 학습하여 반응을 개선하기를 희망긍정적인 고객 피드백에 대한 보상을 제공하는 강화 학습
LLM 결과에서 특정 그룹이나 개인에 대한 편견과 잠재적 차별을 평가. 최소한의 관리 노력으로 LLM 산출물을 평가하길 희망
벤치마크 데이터세트(Benchmark datasets)을 활용
Benchmark datasets: LLM을 포함한 AI 모델들의 성능을 평가하기 위해 특별히 설계된, 미리 준비되고 정제된 데이터 모음. 학생들의 실력을 공정하게 평가하기 위해 표준화된 시험 문제를 사용하는 것과 유사
이미지 분류 작업에 사용되는 기초 모델(FM)의 정확도를 평가하는 전략사전 정의된(pre-defined) 벤치마크 데이터셋을 사용하여 모델의 정확도 측정
검증된 데이터를 사용하면 보다 객관적으로 평가 가능

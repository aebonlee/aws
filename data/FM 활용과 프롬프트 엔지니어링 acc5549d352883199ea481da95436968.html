프롬프트 엔지니어링
  • AI 모델이 더 정확하고 원하는 방식으로 응답을 생성하도록 입력(prompt)을 최적화하는 기술
  • 파인튜닝 없이도 모델 성능을 향상시킬 수 있어 다양한 AI 활용 사례에서 중요
  • 핵심 목표
    • 최소한의 노력으로 최상의 결과 도출 (프롬프트 조정)
    • 명확한 지침과 컨텍스트 제공 (효과적인 프롬프트 설계)
    • 불필요한 응답 제거 및 원하는 출력 형식 유지
  • 기본 원칙
    • 최대한 구체적으로 작성
      • 모든 것을 포괄하는 만능 프롬프트는 없음
    • 원하는 출력 형식과 길이를 포함 (예. 5분 길이, 3단락 등)
    • 모델 재학습, 파인튜닝, RAG를 적용하기 전에 프롬프트부터 조정
      • 모델이 원하는 응답을 생성할 때까지 실험하고 수정
    • 프롬프트 설계의 4대 구성 요소
      • 효과적인 프롬프트는 아래 네 가지 요소를 포함하여 반복적 개선(iterative refinement)을 지원
        구성 요소정의예시
        Task
        (작업)        		모델의 사용 사례를 지정"제품 설명을 바탕으로 마케팅 슬로건을 작성하세요"
        Role
        (역할)        		요구 사항을 효과적으로 충족하기 위해 모델이 맡아야 할 페르소나를 정의"당신은 10년 경력의 카피라이터입니다"
        Response style
        (응답 스타일)        		모델이 따라야 할 어조, 형식, 구조를 설명"캐주얼한 톤, 15자 이내, 한 줄로 작성"
        Success criteria
        (성공 기준)        		모델 출력이 기대치를 충족하는지 평가하기 위한 명확한 지표를 설정"브랜드 키워드를 반드시 포함, 부정적 표현 없음"
    • 시스템 프롬프트 (System Prompt)
      • AI 모델의 역할(Role), 페르소나, 행동 경계(behavioral boundaries)를 사전에 정의하는 지침
      • 사용자의 질문과 별도로 모델이 "어떻게 행동해야 하는지"를 안내
      • 예. "당신은 AWS 전문 기술 지원 엔지니어입니다. 제품 관련 질문에만 답변하고, 정치나 종교 관련 질문에는 답변을 거부하세요."
      • 시스템 프롬프트 vs. 사용자 프롬프트
        구분시스템 프롬프트사용자 프롬프트
        목적모델의 역할, 톤, 제약 조건 설정구체적인 질문이나 요청
        설정 주체개발자/관리자최종 사용자
        변경 빈도앱 배포 시 고정매 대화마다 변경
  • 주요 기법
    이름설명
    Zero-shot prompting예제(example) 없이 질문만으로 AI가 자유롭게 추론하고 답을 생성하도록 유도하는 방식
    모델의 자유로운 추론 유도
    Few-shot prompting도메인 특화 예시나 명시적 지침을 제공하여, 이를 통해 모델이 원하는 결과물의 패턴이나 형식을 참고하는 방식
    특정 형식의 응답 유도
    Prompt chaining복잡한 작업을 여러 개의 작은 하위 작업으로 분할하여 순차적으로 처리하는 기법
    앞선 프롬프트의 결과를 다음 프롬프트의 입력으로 사용하여 과제를 해결
    Positive prompting원하는 결과(특정 톤, 형식, 정보 포함 등)를 얻기 위해 구체적인 조건을 명시하여 모델을 활용하는 방식
    Negative prompting원하지 않는 결과(예: 유해한 콘텐츠, 특정 주제)를 생성하지 않도록 명시적으로 지시하는 기법
    사용자의 프롬프트에서 공격 패턴을 탐지하도록 지시하는 시스템 프롬프트 템플릿도 여기에 해당
    Chain-of-thought prompting모델에게 문제 해결 과정을 논리적으로 생각하고 설명하도록 요구하여 추론의 정확도를 높이는 방식
    예1: 단계별로 생각하라(Let's think step by step)
    예2: Think step by step and walk me through your thinking.
    ReAct
    prompting    		추론(Reasoning)과 행동(Acting)을 반복하며 외부 도구나 API와 상호작용하여 문제를 해결하는 기법
    모델이 사고 과정을 명시적으로 표현하고, 필요한 행동을 취한 후 그 결과를 바탕으로 다시 추론하는 순환 구조
    예: Thought: 재고 확인이 필요함 → Action: inventory_API 호출 → Observation: 재고 5개 → Thought: 충분한 재고 확인
    Message History이전 대화 기록을 프롬프트에 포함하여 문맥을 유지하고 일관된 답변을 얻는 방식
    문맥 유지 및 대화 흐름 개선
    대부분의 챗봇 서비스는 자동으로 적용하지만, API 사용시에는 대화 기록을 전달해야 함
Amazon Bedrock
  • 완전 관리형 서버리스 생성형 AI 서비스
  • 여러 AI 기업들의 기초 모델(FM)을 단일 API를 통해 제공
    • 텍스트 생성 모델, 이미지 생성 모델, 임베딩 생성 모델을 사용한 만큼한 지불(Pay-as-you-go)
    • 기업의 프라이빗 데이터로 모델 커스터마이징 가능
    • Amazon Nova: AWS의 자체 차세대 멀티모달 파운데이션 모델 제품군
      모델특징
      Nova Micro텍스트 전용(Text-only) 모델로, 이미지·비디오 입력을 지원하지 않음
      - Nova 제품군 중 가장 낮은 지연 시간과 비용 제공
      - 텍스트 요약, 간단한 Q&A, 분류 등 텍스트만으로 충분한 작업에 적합
      Nova Lite- 초저지연(Low latency) 및 저비용 처리에 최적화된 모델 (텍스트+이미지+비디오 지원)
      - 간단한 챗봇, 요약, 대규모 텍스트 처리 등 속도와 가성비가 중요한 작업에 적합
      Nova Pro- 복잡한 추론(Reasoning), 코딩, 수학적 문제 해결 등 고성능이 요구되는 작업에 최적화
      - 정확도와 속도의 균형이 뛰어나며, 다양한 도메인의 범용적인 업무 수행 가능
      Nova Canvas- 텍스트 프롬프트를 통해 전문가 수준의 고품질 이미지를 생성 및 편집하는 모델
      - 로고 디자인, 마케팅 자산 생성, 이미지 인페인팅/아웃페인팅 지원
      Nova Reel- 텍스트를 입력하여 고화질 비디오를 생성하는 모델
      - 소셜 미디어 콘텐츠, 광고 영상, 시각 효과 등 영상 제작 업무에 특화
  • 엄격한 데이터 격리 정책으로 완전한 기밀성 보장
    • 사용자의 입력 프롬프트와 모델 출력을 서드파티 모델 제공업체(Anthropic, AI21 Labs 등)와 절대 공유하지 않음
  • 과금 모델
    모델핵심 특징 설명
    온디맨드
    (On-Demand)    		Flexibility
    사용한 만큼 지불    		- 별도의 약정 없이 사용한 만큼만 비용을 지불하는 유연한 모델
    - 입출력 데이터의 양(토큰 수)과 사용한 파운데이션 모델(FM) 종류에 따라 비용이 책정
    - 예산이 제한적이거나 사용량을 예측하기 어려운 경우에 가장 적합
    배치
    (Batch)    		Cost-efficiency
    비동기 워크로드 할인    		- 즉각적인 응답이 필요 없는 대량 처리 작업에 대해 토큰당 요금을 50% 할인
    - 주간 카탈로그 업데이트, 대량 합성 데이터 생성 등 예약된 비긴급 작업에 가장 비용 효율적
    - 비동기적으로 처리되므로 처리 완료 시간은 보장되지 않음
    프로비저닝된 처리량
    (Provisioned Throughput)    		Performance
    안정적인 성능 보장    		- 특정 기간(1개월 또는 6개월) 동안 일정 수준의 처리량을 약정하여 구매
    -     대규모의 안정적인 처리가 필요하거나, 일관된 지연 시간이 중요한 프로덕션 환경에 적합
    - 기본 비용은 매우 비쌈. 장기 약정을 통해 온디맨드보다 저렴하게 사용 가능
  • 주요 기능
    기능설명
    모델 커스터마이징
    (Model
    Customization)    		- 기존의 기초 모델(FM)을 특정 요구사항에 맞게 조정하는 기능 (예: 기업 특화 데이터로 성능 최적화)
    - fine-tuning은 JSONL(JSON Lines) 형식의 입력-출력 쌍(labeled data) 필요
    - Amazon Bedrock에서 커스텀 모델을 사용하려면 프로비저닝된 처리량을 필수로 사용해야 함
    - AWS가 각 모델을 위한 전용 컴퓨팅 리소스를 미리 할당해야 하기 때문
    - Custom Model Import: Amazon SageMaker AI 등 외부에서 훈련한 모델을 Bedrock으로 가져와서 배포할 수 있는 기능
    - SageMaker에서 fine-tuning → Bedrock으로 import → Bedrock API로 서빙
    - Bedrock에서 직접 지원하지 않는 고급 커스터마이징이 필요할 때 활용
    프롬프트 관리
    (Prompt
    Management)    		- 프롬프트를 생성, 테스트, 버전 관리 및 배포할 수 있는 중앙 집중식 도구
    - 다양한 모델 간 프롬프트 성능 비교 및 최적화 지원
    - 프롬프트 변수(Prompt Variables): {{input}} 등의 변수를 사용하여 재사용 가능한 템플릿 생성
    - 도구 구성(Tools configuration): 프롬프트 내에서 외부 API 호출 정의(Function calling)를 설정하여 외부 시스템과 상호작용 가능
    에이전트
    (Agent)    		- FM에 외부 데이터 소스 연결, API 호출, 다단계 추론, 응답 우선순위 결정 등 자율적 태스크 수행 능력 부여
    - 반복적인 작업을 자동화하고 복잡한 워크플로우를 오케스트레이션
    - 단순한 텍스트 생성(단일 FM)으로는 해결할 수 없는 외부 시스템과의 상호작용이 필요한 작업에 적합
    - 여러 단계의 작업을 순차적으로 실행하고, 필요한 경우 외부 도구를 사용 (예: API 호출)
    - 예1. 고객 문의 자동 처리 및 응답
    - 예2. OpenSearch에서 고객 데이터 검색 → CSV 파일 생성 → S3에 업로드를 하나의 자연어 요청으로 처리
    - Model Context Protocol (MCP) 지원
    지식 기반
    (Knowledge Base)    		- AI 모델이 참조할 수 있는 구조화된 지식 저장소
    - 외부 문서, 기업 데이터, 정책 문서, 최신 데이터 등을 포함
    - 예1. 챗봇이 고객 질문에 답변할 때 제품 매뉴얼 참조
    - 예2. AI 모델이 회사 정책에 맞는 응답 생성 시 내부 규정 확인
    - 예3. 특정 도메인의 전문 지식이 필요한 작업 수행 시 관련 자료 활용
    워터마크 감지
    (Watermark
    Detection)    		- Amazon Titan Image Generator가 생성한 이미지를 식별하는 기능
    - 이미지의 진위 여부를 확인하고, AI가 생성한 이미지와 실제 이미지를 구분
    가드레일
    (Guardrails)    		- Bedrock이 생성하는 콘텐츠 통제. 규정 위반 여부에 대한 사용자 입력 모니터링. 책임감 있는 AI(Responsible AI)를 보장
    - 주요 정책 유형
    - 콘텐츠 필터(Content Filters): 혐오 발언, 모욕, 성적 콘텐츠, 폭력 등 유해 카테고리별 필터링 강도를 설정
    - 거부 주제(Denied Topics): 관리자가 정의한 특정 주제 영역의 논의를 차단 (예: 투자/법률 조언, 의료/ 진단, 정치)
    - 프롬프트 공격(Prompt Attacks): 프롬프트 인젝션 및 탈옥(Jailbreak) 시도를 탐지하고 차단. AI의 내장 안전 기능을 우회하려는 시도를 방지
    - 민감 정보 필터(Sensitive Information Filters): 입출력에서 PII(개인 식별 정보) 또는 민감 데이터를 탐지하고 마스킹/삭제
    - 문맥 기반 근거 확인(Contextual Grounding Check): 모델 응답이 제공된 소스 자료에 근거하는지 검증. 환각(Hallucination) 방지에 활용
    - 단어 필터(Word filters): 공격적인 용어를 감지하고 차단 (예: 경쟁사 언급 금지)
    모델 평가
    (Model Evaluation)    		- 사용 사례에 가장 적합한 기초 모델을 평가, 비교, 선택할 수 있게 해주는 기능
    - 인적 기반(human-based) 모델 평가 작업 지원 (예: 전문가 팀이 모델 출력을 직접 평가하고 비교)
    호출 로깅
    (Invocation Logging)    		- 활성화할 경우, 모델 호출 로그를 저장
    - 모델 입력 및 출력 데이터를 모니터링 가능
    배치 추론
    (Batch
    Inference)    		- 대량의 프롬프트를 비동기적(asynchronously)으로 처리하는 기능
    - 즉각적인 응답이 필요하지 않은 대규모 워크로드에 적합
    - 예. 주간 보고서, 수천 개의 제품 설명 생성, 합성 데이터 응답 대량 생성
    - 실시간 추론 대비 토큰 요금 50% 할인

  • Generative AI Security Scoping Matrix(생성형 AI 보안 범위 정의 매트릭스)
    • 생성형 AI 사용 사례를 분류하고, AI 솔루션에 필요한 소유권 수준을 평가하거나 보안 문제의 우선순위를 정하기 위한 프레임워크
      • 마치 자동차의 안전 등급을 매기는 것과 비슷. 일반 승용차, 어린이 통학버스, 현금 수송차는 각각 다른 수준의 안전 장치가 필요한 것처럼, AI 시스템도 용도에 따라 적절한 보안 수준이 필요
    • 기업이 생성형 AI를 도입할 때 "어떤 수준의 보안이 필요한지", "어떤 위험이 있는지", "어떤 보호 조치를 먼저 해야 하는지"를 체계적으로 판단 가능
    • 보안 책임(the ownership of security responsibilities)의 크기 비교
      접근 방식설명
      생성형 AI 기능이 내장된 서드파티 기업용 애플리케이션 사용가장 기본적인 접근 방식(가장 낮은 책임)
      이미 만들어진 AI 기능이 포함된 소프트웨어를 그대로 사용하는 것
      기존 서드파티 생성형 AI 기초 모델(FM)을 활용한 애플리케이션 구축한 단계 더 깊이 들어가는 방식
      기존의 AI 모델을 기반으로 자신만의 애플리케이션을 만드는 것(예. GPT 모델을 활용해 고객 서비스 챗봇을 구현)
      기업 특화 데이터를 사용하여 기존 서드파티 생성형 AI 기초 모델(FM)을 미세 조정기존 AI 모델을 자신의 필요에 맞게 추가 학습시키는 방식
      (예. 법률 회사가 기존 언어 모델을 법률 문서로 추가 학습시켜 법률 전문 AI로 만드는 경우)
      고객이 보유한 특정 데이터를 사용하여 처음부터 생성형 AI 모델을 구축하고 훈련가장 깊은 수준의 접근 방식(가장 높은 책임)
      AI 모델을 완전히 처음부터 만드는 것
